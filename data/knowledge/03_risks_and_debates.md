# 争议与风险：Transformer 与大模型项目的常见挑战

## 争议 1：模型规模越大是否一定更好

支持“大模型优先”观点：
1. 泛化能力与复杂推理能力通常更强
2. 少样本与零样本表现更稳定

反对观点：
1. 推理成本高，部署门槛高
2. 在垂直任务上，小模型微调可能更高性价比

折中方案：  
基座模型按成本分层，先用中等规模模型验证场景，再按指标升级。

## 争议 2：开源模型还是闭源 API

支持开源模型观点：
1. 可私有化部署，数据可控
2. 可二次训练和深度定制

支持闭源 API 观点：
1. 迭代快，工程接入成本低
2. 通常具备更完善的服务稳定性

折中方案：  
研发阶段可先用 API 验证价值，生产阶段按合规与成本决定是否迁移开源方案。

## 风险清单

1. 长上下文性能退化风险  
影响：长文档问答准确率下降。  
触发条件：位置编码外推失败或注意力计算受限。

2. 幻觉与事实错误风险  
影响：输出内容听起来合理但不正确。  
触发条件：缺乏检索增强、约束解码或事实校验流程。

3. 数据与版权合规风险  
影响：训练数据来源争议或泄露敏感信息。  
触发条件：数据采集与清洗缺少合规审查。

4. 评估不充分风险  
影响：离线看起来有效，上线后表现不稳定。  
触发条件：缺少分场景评估和持续监控。

## 适用前提

1. 团队具备基础深度学习与工程化能力
2. 业务场景能提供明确评估指标
3. 能接受“模型能力、成本、时延”三者间的权衡
